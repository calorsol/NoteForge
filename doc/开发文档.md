## 实现现状（2026-06-25 更新）

**M1 ~ M6 已全部实现，MVP 可运行。** 后端 10 个测试全部通过，前端构建通过，并已做端到端冒烟验证（注册→建资料→补录历史日期→建文档→自动保存→越权/无 token 均返回正确状态码）。

与本文档初始方案的两处**有意偏差**（为减少依赖与构建复杂度）：
1. **未使用 Tailwind**，改为手写 CSS（`src/client/src/styles/index.css`），统一设计变量，效果同样克制美观。
2. **未使用 `@uiw/react-md-editor`**，改为自写零依赖 Markdown 渲染器（`src/client/src/markdown.ts`）+ `textarea` 编辑 + 实时预览分栏。好处是无需额外安装、构建稳定；代价是 Markdown 语法支持为常用子集（标题/粗斜体/行内代码/代码块/有序无序列表/引用/分割线/链接）。若日后需要表格、图片等完整语法，可再引入 `react-markdown`。

> 已知遗留：`db.ts` 引入 `better-sqlite3` 时 `tsc --noEmit` 报缺少类型声明（不影响 tsx 运行与 vite 构建）。如需消除，`npm i -D @types/better-sqlite3` 即可。

### UI 改版（在 MVP 基础上，按用户评审定稿）

- **资料库页**：左侧日期由扁平列表改为 **年 → 月 → 日级联**（先选年再选月，下方列出当月有资料的日期带条数）；顶部横向资料 Tab 改为 **第二栏纵向列表 + 搜索**（应对一天大量资料）；整体为「日期 / 资料列表 / 编辑」三栏。
- **写作中心**：左侧新增 **资料抽屉**——手风琴卡片（默认露摘要、点开看全文渲染、可多条同时展开、独立滚轮滚动），顶部日期选择 + **「+ 添加资料」内联表单**（不离开写作页即可新建资料）；中间写作区支持 **文本 / Markdown 模式切换**（文本模式隐藏预览并铺满）；右侧预览 **可收起/展开**；多文档通过顶部 **文档下拉**切换/新建。
- **删除确认**：统一改为自定义 `ConfirmDialog` 弹窗（替代浏览器原生 `confirm`），资料与文档删除共用。
- **图标**：未引第三方图标库，使用 `src/client/src/components/icons.tsx` 内置内联 SVG。
- 以上均已在浏览器实测渲染与交互通过。

### 写作中心增强（第二轮评审）

- **外观切换**：新增 `src/client/src/appearance.ts` —— 字体（宋体/楷体/黑体/仿宋/等宽，优先系统中文字体栈，零网络依赖）、字号（小/中/大）、背景主题（米白/纯白/护眼/暖褐/夜间，含深色）。工具栏「Aa 外观」弹层切换，选择持久化到 `localStorage`（`nf_font` / `nf_size` / `nf_bg`），应用于写作区与预览。
- **按文档记忆模式**：Markdown / 文本 模式按文档保存于 `localStorage`（`nf_mode_<docId>`），打开文档时读取，删除文档时清除；新建文档默认 Markdown。
- **可拖拽宽度**：资料抽屉与预览栏之间各有一根拖拽分隔条（`ResizeHandle`，pointer 事件），宽度持久化（`nf_drawerW` / `nf_previewW`）；编辑区设 `min-width` 防止被挤没。
- **抽屉吸顶**：抽屉头部与「+ 添加资料」栏固定在顶部（在滚动容器之外），仅资料卡片列表滚动；每张卡片的标题栏 `position: sticky` 吸顶，展开后滚动长内容时仍可随时收起。

### 资料删除交互（第三轮评审）

- **可见入口**：编辑区标题行右上角的垃圾桶图标（替代原先底部的「删除资料」按钮），点击 → 确认弹窗 → 删除（安全入口）。
- **右键快删**：资料列表项支持 `onContextMenu` 右键菜单（`.context-menu`，`position: fixed` 跟随光标），点「删除」**直接删除、不弹确认**（快捷入口）。右键同时会选中该项；点空白处或再次右键关闭菜单。
- 两条路径共用核心 `deleteMaterialById(id)`；均已浏览器实测。
- **写作中心资料抽屉**同样支持右键菜单删除（`MaterialDrawer` 内 `onContextMenu` + `.context-menu`），右键 → 删除 → 直接删除、不弹确认，与资料库一致；已浏览器实测。

## 0. 给 AI 实现者的总则（务必先读）

1. **数据隔离是硬约束**：每个数据接口都必须从 JWT 解析出 `userId`，并只操作该用户的数据。任何 `materials` / `documents` 查询都要带 `WHERE user_id = ?`。禁止信任前端传来的 `user_id`。
2. **密码永不明文**：注册时 `bcrypt.hash`，登录时 `bcrypt.compare`。数据库只存 `password_hash`。
3. **接口契约不要随意改**：第 5 节的 API 是前后端约定，改动需同步前端。
4. **保持简单**：这是单人工具，不要引入 Redis、消息队列、微服务等。SQLite 足够。
5. **先跑通 MVP**：按第 8 节里程碑顺序实现，每步可独立验证。
6. **Windows 环境**：开发机为 Windows + PowerShell，命令需兼容（npm 脚本天然跨平台即可）。

## 1. 技术栈

### 前端
- **React 18 + Vite + TypeScript** —— 快速、现代。
- **Tailwind CSS** —— 实现"漂亮、克制"的界面。
- **React Router** —— 路由（登录/注册/资料库/写作中心）。
- **Markdown 编辑/预览**：`@uiw/react-md-editor`（自带编辑+实时预览，开箱即用），或 `react-markdown` + `textarea` 自行组合。优先用 `@uiw/react-md-editor`。
- **请求**：原生 `fetch` 封装一个 `api.ts`（带 JWT header）。

### 后端
- **Node.js + Express + TypeScript**
- **SQLite**，驱动用 **better-sqlite3**（同步 API，简单可靠，单文件）。
- **认证**：`bcrypt`（密码哈希）+ `jsonwebtoken`（签发/校验 JWT）。
- **CORS**：开发期前后端分端口，用 `cors` 放行；生产可由后端托管前端静态文件。
- **校验**：`zod` 校验请求体（轻量）。

### 为什么这样选
- SQLite 单文件零配置，复制即备份，最契合"个人工具 + 易部署"。
- better-sqlite3 同步 API，代码直白，没有回调地狱。
- JWT 无状态，前端存 token 即可，省去会话存储。

## 3. 数据模型（SQLite 表结构）

```sql
-- 用户表
CREATE TABLE IF NOT EXISTS users (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  username      TEXT    NOT NULL UNIQUE,
  password_hash TEXT    NOT NULL,
  created_at    TEXT    NOT NULL DEFAULT (datetime('now'))
);

-- 资料表
CREATE TABLE IF NOT EXISTS materials (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  user_id     INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  day         TEXT    NOT NULL,                 -- 采集日期，格式 'YYYY-MM-DD'
  title       TEXT    NOT NULL,
  content     TEXT    NOT NULL DEFAULT '',
  created_at  TEXT    NOT NULL DEFAULT (datetime('now')),
  updated_at  TEXT    NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX IF NOT EXISTS idx_materials_user_day ON materials(user_id, day);

-- 文档表
CREATE TABLE IF NOT EXISTS documents (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  user_id     INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  title       TEXT    NOT NULL DEFAULT '无标题文档',
  content     TEXT    NOT NULL DEFAULT '',
  created_at  TEXT    NOT NULL DEFAULT (datetime('now')),
  updated_at  TEXT    NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX IF NOT EXISTS idx_documents_user ON documents(user_id, updated_at);
```

**设计说明**
- "日期 Tab"不需要单独的表：对 `materials` 按 `day` 分组即可（`SELECT DISTINCT day ...`）。
- `day` 用字符串 `YYYY-MM-DD`，便于按日分组、排序，避免时区坑。
- 删除用户时级联删除其资料与文档（`ON DELETE CASCADE`，需开启 `PRAGMA foreign_keys = ON`）。

## 4. 认证设计

- 注册：校验用户名唯一 → `bcrypt.hash(password, 10)` → 入库 → 直接签发 JWT 返回（自动登录）。
- 登录：取用户 → `bcrypt.compare` → 通过则签发 JWT。
- **JWT**：`payload = { userId, username }`，`expiresIn: '7d'`，密钥从环境变量 `JWT_SECRET` 读取（无则用默认开发密钥并告警）。
- 前端把 token 存 `localStorage`，每次请求加 `Authorization: Bearer <token>`。
- `requireAuth` 中间件：解析 token 失败/过期 → 返回 `401`；成功则把 `req.userId` 挂上供后续使用。

## 5. API 契约

所有接口前缀 `/api`。除注册/登录外都需 `Authorization: Bearer <token>`。
统一错误格式：`{ "error": "可读的错误信息" }`，HTTP 状态码语义化（400/401/403/404/409/500）。

### 5.1 认证 `/api/auth`

| 方法 | 路径 | 请求体 | 成功响应 |
|------|------|--------|----------|
| POST | `/api/auth/register` | `{ username, password }` | `201` `{ token, user: { id, username } }` |
| POST | `/api/auth/login` | `{ username, password }` | `200` `{ token, user: { id, username } }` |
| GET  | `/api/auth/me` | — | `200` `{ user: { id, username } }`（用于刷新页面后校验登录态） |

错误：用户名占用 → `409`；用户名/密码不符 → `401`；字段非法 → `400`。

### 5.2 资料 `/api/materials`

| 方法 | 路径 | 说明 | 响应 |
|------|------|------|------|
| GET  | `/api/materials/days` | 当前用户所有有资料的日期 + 每天条数（倒序），供年/月/日级联与条数徽标使用。 | `{ days: [ {day:"2026-06-25", count:3}, {day:"2026-06-24", count:5} ] }` |
| GET  | `/api/materials?day=YYYY-MM-DD` | 某天的全部资料（按创建时间正序）。`day` 省略则默认今天。 | `{ materials: [ {id,day,title,content,updated_at}, ... ] }` |
| POST | `/api/materials` | 新增资料。`{ day?, title, content? }`，`day` 省略默认今天。 | `201` `{ material }` |
| PUT  | `/api/materials/:id` | 更新资料 `{ title?, content?, day? }`，更新 `updated_at`。 | `200` `{ material }` |
| DELETE | `/api/materials/:id` | 删除资料。 | `204` |

> 所有操作均隐含 `WHERE user_id = req.userId`；`:id` 不属于当前用户 → `404`（不暴露存在性）。

### 5.3 文档 `/api/documents`

| 方法 | 路径 | 说明 | 响应 |
|------|------|------|------|
| GET  | `/api/documents` | 当前用户文档列表（不含正文或含正文均可，建议列表只回 `id,title,updated_at`）。按 `updated_at` 倒序。 | `{ documents: [ {id,title,updated_at}, ... ] }` |
| GET  | `/api/documents/:id` | 单篇完整内容。 | `{ document: {id,title,content,updated_at} }` |
| POST | `/api/documents` | 新建。`{ title?, content? }`。 | `201` `{ document }` |
| PUT  | `/api/documents/:id` | 更新（自动保存调用此接口）`{ title?, content? }`。 | `200` `{ document }` |
| DELETE | `/api/documents/:id` | 删除。 | `204` |

## 6. 前端实现要点

### 6.1 登录态
- `AuthContext` 持有 `{ token, user }`，从 `localStorage` 初始化。
- 应用启动若有 token，调 `GET /api/auth/me` 校验；失效则清除并跳登录。
- 路由守卫：未登录访问受保护页 → 重定向 `/login`。

### 6.2 资料库页面（MaterialsPage）
- 进入时：`GET /api/materials/days` 得到日期列表 → 渲染第一级 Tab，默认选中今天（不在列表里也展示"今天"）。
- 选中某天：`GET /api/materials?day=...` → 渲染第二级资料 Tab，默认选中第一份。
- "+ 新增资料"：在当前选中日期下创建，成功后刷新资料 Tab 并选中新建项。
- 资料编辑：标题 + 内容（Markdown），「保存」调 `PUT`。
- 删除：二次确认 → `DELETE` → 刷新。
- 空状态引导。

### 6.3 写作中心（WritePage）
- 左侧 `DocumentList`：`GET /api/documents` 渲染，按更新时间倒序，显示标题+时间。
- "+ 新建"：`POST /api/documents` → 选中新文档进入编辑。
- 右侧 `MarkdownEditor`（封装 `@uiw/react-md-editor`）：编辑标题 + 正文，实时预览，字数统计。
- **自动保存**：对标题/正文变更做 **debounce 1.5s** → `PUT /api/documents/:id`；显示"保存中…/已保存 HH:MM"。也提供手动保存按钮。
- 删除：二次确认 → `DELETE`。

### 6.4 视觉风格（"漂亮"的落地标准）
- 大量留白，正文阅读宽度约 `680–760px` 居中。
- 字体：正文衬线或高可读无衬线，行高 1.7+。
- 配色克制：浅色背景 + 一个主题强调色；Tab 选中态清晰。
- 编辑器聚焦时弱化周边干扰（专注感）。

## 8. 实施里程碑（建议按序）

> 每个里程碑结束都应能独立运行/验证。

**M1 — 工程骨架**
- 建 `src/server` 与 `src/client` 两端可分别启动；前端能请求到后端 `/api/health` 返回 ok。

**M4 — 资料库**
- 后端 materials 五个接口 + 前端两级 Tab + 增删改查 + 空状态。

**M5 — 写作中心**
- 后端 documents 接口 + 前端文档列表 + Markdown 编辑器 + debounce 自动保存。

**M6 — 打磨**
- Tailwind 视觉打磨、加载/错误态、二次确认、字数统计、移动端基本可用。

## 10. 已知约定 / 注意事项
- `better-sqlite3` 为原生模块，Windows 安装需要构建工具链；若安装失败，可改用 `node:sqlite`（Node 22+）或 `sql.js`。优先 better-sqlite3。
- 时间一律用服务器本地时间生成 `day`（`YYYY-MM-DD`）；如需严格时区，后续再统一。
- `.gitignore` 应忽略 `node_modules/`、`src/server/data/*.db`、`.env`。
