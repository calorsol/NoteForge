## 2. 部署架构

### 2.1 应用本身

NoteForge 是 **React（Vite）前端 + Express + SQLite 后端**的全栈应用：

- 前端请求走相对路径 `/api/*`，因此前端与后端**必须同源**。
- 关键改造（本期）：生产环境下 **Express 同时托管前端构建产物 `src/client/dist`**，并对非 `/api` 路由回退到 `index.html`（SPA fallback）。这样**单个端口（4000）同时提供 UI 和 API**，既支持「IP:端口」直连，也让后续接 nginx/域名变得简单。
  - 实现见 `src/server/app.ts`：在所有 `/api` 路由注册之后，`express.static(clientDist)` + 一个跳过 `/api` 的兜底中间件返回 `index.html`。用 `fs.existsSync(clientDist)` 保护，dev 与测试环境（无 dist）下自动跳过。

### 2.2 服务器环境（沿用既有约定）

该服务器整套基础设施都跑在 **Docker**，由 `/home/web/docker-compose.yml` 统一编排：

- `nginx`（`nginx:alpine`，**host 网络模式**）—— 全站统一反向代理 / Web 服务器，占用宿主机 80/443。
- `php` / `php74` / `mysql` / `redis` —— 服务于该机其它 PHP 站点。

每个站点一份 nginx 配置放在 `/home/web/conf.d/*.conf`；证书在 `/home/web/certs/`；ACME 验证根目录 `/home/web/letsencrypt/`。

NoteForge 是该机的**第一个 Node 应用**，采用**独立的一套 compose**，与上述基础设施隔离，互不影响。

### 2.3 NoteForge 的容器化

目录结构（`/home/web/noteforge/`）：

```
/home/web/noteforge/
├── docker-compose.yml   # 应用编排（容器名 noteforge，发布 4000）
├── Dockerfile           # 多阶段构建
├── .dockerignore
├── .env                 # JWT_SECRET（仅服务器有，随机生成，不进 git/镜像）
├── data/                # SQLite 持久化目录（bind mount）
└── src/ ...             # 应用源码
```

`docker-compose.yml`：

```yaml
services:
  noteforge:
    build: .
    container_name: noteforge
    restart: always
    ports:
      - "4000:4000"
    environment:
      - PORT=4000
      - NODE_ENV=production
      - JWT_SECRET=${JWT_SECRET}   # 从同目录 .env 读取
    volumes:
      - ./data:/app/src/server/data  # SQLite 持久化，重建容器不丢数据
```

`Dockerfile`（要点）：

- 多阶段，基底 **`node:22-bookworm-slim`（Debian）**——选 Debian 而非 Alpine，是为了让原生模块 `better-sqlite3`、`bcrypt` 顺利安装/编译。
- build 阶段：`npm ci` → `npm run build:client`（产出 `src/client/dist`）。
- runtime 阶段：拷贝 `node_modules` + `src` + 配置，`NODE_ENV=production`，以 `npx tsx src/server/index.ts` 运行（无需预编译 TS）。

> ⚠️ 原生模块坑：`better-sqlite3` / `bcrypt` 含平台相关的二进制，**不能把 Windows 本地的 `node_modules` 拷到 Linux**，必须在 Linux（容器内）重新安装。`.dockerignore` 已排除 `node_modules`、`dist`、`.env`、`*.db`、日志等。

### 2.4 数据持久化

SQLite 数据库文件位于宿主机 `/home/web/noteforge/data/noteforge.db`，通过 bind mount 映射到容器 `/app/src/server/data`。**重建/升级容器不会丢数据。**

## 3. 域名与 HTTPS

### 3.1 DNS

`noteforge.868601.xyz` 在 Cloudflare，目前为 **DNS-only（灰云）**，即直接解析到源站 `64.83.35.146`，**不经过 Cloudflare 代理**。因此 TLS 在源站终止，需要源站持有受信任证书（与该机其它站点 echotype/cliapi 一致）。

### 3.2 证书（Let's Encrypt，自动续期）

- 工具：服务器上的 **acme.sh**（`/root/.acme.sh/`）。
- 签发方式：HTTP-01 webroot（`-w /home/web/letsencrypt`），CA 指定 `--server letsencrypt`，`--keylength ec-256`。
- 安装位置：`/home/web/certs/noteforge.868601.xyz_cert.pem` 与 `_key.pem`（命名与其它站点一致）。
- 续期：acme.sh 每日 `cron` 自动检查；续期后执行 reloadcmd `docker exec nginx nginx -s reload` 重载 nginx。
- 当前证书有效期至 **2026-09-23**，预计续期窗口 2026-08-25 前后。

### 3.3 nginx 反向代理

站点配置：`/home/web/conf.d/noteforge.868601.xyz.conf`（项目内留有副本 `deploy/noteforge.868601.xyz.conf`）。逻辑：

- `:80` → 301 跳转到 `https://`（保留 `/.well-known/acme-challenge/` 供续期）。
- `:443` ssl → `proxy_pass http://127.0.0.1:4000`（nginx 是 host 网络模式，可直接访问宿主机端口）。

> 改完 nginx 配置后生效命令：`docker exec nginx nginx -t && docker exec nginx nginx -s reload`

## 5. 一次完整部署/更新流程（操作手册）

> 适用于「本地改完代码 → 上线」。所有命令在本地项目根目录 `D:\AI\WebSite\14_NoteForge` 与服务器之间进行。

**1）本地打包**（排除依赖/构建产物/数据库/密钥）：

```bash
tar --exclude='./node_modules' --exclude='./.git' --exclude='./.claude' \
    --exclude='./.playwright-mcp' --exclude='./dist' --exclude='./src/client/dist' \
    --exclude='./doc' --exclude='*.log' --exclude='./.env' \
    --exclude='./src/server/data/noteforge.db' --exclude='./src/server/data/test-artifacts' \
    -czf /tmp/noteforge.tar.gz .
```

**2）上传并解压**（解压不会覆盖服务器的 `.env` 和 `data/`，因为这两者不在包里）：

```bash
scp /tmp/noteforge.tar.gz root@64.83.35.146:/home/web/noteforge/
ssh root@64.83.35.146 "cd /home/web/noteforge && tar -xzf noteforge.tar.gz && rm noteforge.tar.gz"
```

**3）重建并重启容器**：

```bash
ssh root@64.83.35.146 "cd /home/web/noteforge && docker compose up -d --build"
```

**4）验证**：

```bash
curl -s https://noteforge.868601.xyz/api/health        # 期望 {"ok":true}
curl -s -o /dev/null -w '%{http_code}\n' https://noteforge.868601.xyz/   # 期望 200
```

> 说明：当前部署用 tar+scp，**不依赖 GitHub**（GitHub 仅作源码托管/备份）。若以后想改成服务器 `git pull` 部署亦可，但需在服务器上把目录改成 git 克隆。

## 7. 已知待办与优化建议

按收益/优先级排序：

1. **登录失败文案误导（建议尽快修）**
   - 现象：客户端 `src/client/src/api.ts` 把**所有 401** 都改写成「登录已过期，请重新登录」。导致登录时账号/密码错误（服务端本应返回「用户名或密码错误」）也显示成「登录已过期」，容易误导。
   - 修法：仅当请求**带着 token**（即确实登录过）才视为「登录已过期」并清 token；否则透传服务端真实错误文案。

2. **静态资源缓存（提升二次打开速度）**
   - 现象：带哈希、本可永久缓存的 `/assets/*.js|css` 现在被 Express 以 `Cache-Control: max-age=0` 返回，每次刷新都重新传输（~190KB JS）。
   - 修法：对 `/assets/*` 设长期 `immutable` 缓存，`index.html` 保持不缓存。

3. **Cloudflare 橙云（CDN 加速 + 隐藏源站）**
   - 现状：灰云直连，源站在东京。实测**服务器处理仅 2~14ms，慢在跨网络往返**（无 CDN、无边缘缓存）。
   - 建议：把该 DNS 记录切成**橙云（Proxied）**，SSL/TLS 模式设 **Full (strict)**（源站已有受信任 LE 证书，切完即可正常工作，应用无需改动）。可显著改善「切页面/点按钮」的体感延迟，并隐藏源站 IP。

4. **可选增强**：资料库/文档的全文搜索、Markdown 表格与图片支持、移动端适配、定期自动备份数据库。

## 8. 本次会话流程梳理（时间线）

1. **部署到云服务器**：调研服务器（发现全 Docker 架构）→ 给 Express 增加前端静态托管 → 编写 Dockerfile / compose / .dockerignore → tar+scp 上传到 `/home/web/noteforge` → `docker compose up -d --build` → 验证健康检查、注册/登录、单端口服务 UI+API。
2. **绑定域名 + HTTPS**：确认 `noteforge.868601.xyz` 灰云直连源站 → acme.sh 走 HTTP-01 签发 Let's Encrypt 证书 → 装到 `/home/web/certs` → 新增 nginx vhost（80→443 + 反代 4000）→ 验证 HTTPS、证书受信、自动续期。
3. **性能诊断**：实测定位「卡顿」根因为**网络往返 + 无 CDN + 静态缓存关闭**，服务器本身不慢；并发现部分交互一次点击连发多个串行请求会放大延迟。
4. **「登录已过期」排查**：定位为客户端把所有 401 统一改文案所致（实为账号不存在/密码错误），后端无问题。
5. **本期功能迭代**：资料库阅读/编辑视图 + 字体排版优化；写作中心资料抽屉双击编辑、标题吸顶、右键删除；修复 StrictMode 下点击逻辑。**本地用预览工具完整验证通过。**
6. **提交与上线**：两个提交推送到 GitHub `main` → tar+scp + `docker compose up -d --build` 重新部署（保留 `.env` 与数据库）→ 验证新版生效、数据完好 → 清理调试测试账号，仅保留 `chome`。
